## 〇、前置·接通独立测量总线（第18章）

**这一节是所有观测信号与逃逸率的计分地基，缺了它，下面 12 个信号、四象限运营看板、慢外环全是无源之水。** 第18章把它定位成"耦合两环的总线"：快内环产出的每条 trace 在这里被采集、评估、换算成一个可比的数，慢外环再以这些数为唯一依据去改进。先把这条总线接通、且接得"敢当真理源"，再谈观测什么。

四道闸守住"敢当真理源"这条底线：

1. **测量必须独立于被测对象（量者≠被量者）。** 让 Agent 给自己的输出打分 = 生成者自评 = 自证清白，几乎毫无价值——产生错误的那套知识/推理，和评判错误的那套是同一套，它看不见的盲区恰恰是它会犯的那类错。工程形态：审查者/评估器与生成者**绝不共用同一套知识库**。生成靠相似性检索，审查靠独立规则/标准硬核对，两者失效方式不同，才抓得住对方的错。检验独立性的务实办法：把生成者犯过的真实错误取出来，看审查者能否抓住；抓不住 = 共因失效，只是形式上的审查。

2. **去现场看真产物，不信仪表盘转述、不信子代理自述。** Agent 系统层层嵌套，每经一层信息就被转述/失真一次；仪表盘上的绿点是"一份关于真产物的、被转述了四五道的摘要"。最容易出错的是轻信子代理"已完成/超时/没写成"的汇报——它是"数据"不是"真相"，常见情况是收尾汇报报错而磁盘上内容其实已完整落地。判断一件事是否完成，必须直接核对真实交付物（数据库记录、报告文件、审批流真实状态）。这正是 trace 的真正用途：存下每一步真实输入输出（模型真读到的上下文、真传给工具的参数、工具真返回的结果），细到能复现，让排障者绕过所有转述直接回放现场——否则 trace 只是又一层转述。

3. **裁判的职责是找反例，不是收集通过证据。** 收集通过证据是所有测量系统的默认倾向（人天然确认预期、测试集不知不觉只剩正例）。本节主张证伪：竭力去找"这个输出其实是错的"的证据，翻遍所有已知失败模式都找不到反例，才暂时接受它对。落到评估集：**不能只堆正例，历史反例和边界样本要有固定权重**，把 Agent 真实犯过的每个错固化成永久反例——测量系统应越测越严苛，反例越积越厚。

4. **测量系统自己也要被测量（MSA 量具自标定）。** 尺子若是弯的，量得越认真偏差越稳定。一套独立的、看真产物的、专门找反例的测量系统，自身仍可能"不准"，因此它必须像工业量具一样被自标定：定期用已知对错的标定样本回灌，验证它判对/判错的能力是否仍在基线上；同时为"何时不敢下结论"配统计功效门槛，样本不足时诚实降级标注为"观测样本不足、待积累"，而不是把随机波动当信号输出一个站不住的结论。**不可自证的 oracle**——计分的依据不能来自被计分者自己。

> 缺了这条总线，下面所有观测信号与逃逸率都失去可信地基；慢外环（第21章 DMAIC）就只能凭主观印象判断"最近错得有点多"，而非凭真实测量判断"它跳出了控制带"。

## 一、表19-1：生产环境必须上报的12个观测信号（第19章 19.3）

第19章 19.3 的朴素原则："**没测的量，一定会漂。**"系统的可控范围＝它所感知信号的集合；未测量的维度就是失控的维度。所以信号要铺在"会漂的地方"，而不是"好测的地方"——延迟和错误率好测，却未必能暴露语义失效。下表逐字对应书中**表19-1**，每个信号偏离基线就当场推到运营看板、触发对应告警或审查。

| # | 信号 | 它在盯什么"漂" | 漂了通常意味着 |
|---|------|--------------|--------------|
| 1 | 任务成功率 | 端到端真正交付成功的比例 | 最核心的质量信号；它垮了，别的指标都没意义 |
| 2 | 节点失败率 | 链路上每一步各自的失败率 | 定位"故障热点步"，某步突然从 2% 跳到 15%，多半是它依赖的工具或数据源变了 |
| 3 | 重试率 | 自动重试被触发的频率 | 飙到两成以上＝在用算力成本掩盖底层质量问题（见下文"花钱买假象"） |
| 4 | 人工接管率 | 任务被升级给人处理的频率 | 衡量自主能力最直接的尺子；连周上升，常指向知识库该更新或策略该调 |
| 5 | 护栏拦截率 | 输入/输出护栏被触发的频率 | 过低＝护栏太松形同虚设；异常过高＝护栏太激进，或系统正被有组织地攻击 |
| 6 | 单次任务词元成本 | 完成一次完整任务烧掉的词元与对应货币 | 突然翻倍，通常指向上下文膨胀或模型路由策略失效 |
| 7 | 知识命中率 | 检索召回的知识里真正被引用的比例 | 过低（不足三成）＝在为一大堆无用上下文付费，检索策略该优化 |
| 8 | 缓存命中率 | 精确与语义缓存的综合命中比例 | 突然下降，通常意味着业务模式变了 |
| 9 | 模型降级率 | 因旗舰模型超时/过载被降级到备用模型的比例 | 超过一成，旗舰模型的容量规划该重估 |
| 10 | 平均审批耗时 | 从发起审批到人完成审批的平均时长 | 持续超阈值＝"审批疲劳"的先行指标，该挂超时升级链 |
| 11 | 错误恢复率 | 撞到报错后靠读日志自我修复的成功率 | 衡量韧性；恢复失败率走高＝系统在硬撑 |
| 12 | 平均端到端时延 | 从发起到交付最终结果的总耗时 | 突然飙升，通常指向下游依赖性能退化 |

> **重试率最容易被读反**（第19章 19.3）：直觉上重试是韧性，但系统性偏高时它从"韧性"翻成"假象"——在花算力的钱买一个"系统很健康"的假象。底层那个"报表格式时好时坏"的真问题被重试掩盖，永远浮不上来。成熟看板会把**重试率和单次任务成本并排放**：成功率持平、成本却悄悄爬升时，十有八九是重试在为某个失效环节持续承担成本。盯重试率，盯的不是"它有没有最终成功"，而是"它有没有在用成本掩盖底层质量缺陷"。

### Trace分辨率要求（第19章 19.2）

第19章 19.2 把 Trace 定位为"可回放的执行审计记录"，故障定位从三个角度可解：决策依据回放、工具调用复盘、上下文快照。必须能在1分钟内调出任意出错任务的：

- **可披露推理摘要 + 决策依据**：模型该步可对外披露的推理摘要，以及它凭什么选了这个工具（**不要求采集完整内部思维链**，详见下方扩展注）
- **证据引用**：该步引用到的具体证据/知识片段
- **工具真实入参出参**：传给工具的真实参数、工具真实返回的数据
- **中间状态**：定位错误转折点出现在哪一节
- **上下文注入快照**：检索模块实际注入了哪几篇文档片段（排查"把两份不同失效模式的旧案例混进同一窗口"这类证据污染）

> **本地实现建议（书把完整 CoT 放在脚注的语义可观测概念里）**：第19章 19.1/19.2 要求的是**可披露推理摘要**，并在脚注 [1] 处说明语义可观测层（思维链、工具决策依据、上下文注入快照）与思维链（CoT）的边界——书正文不把"采集完整内部思维链"当硬要求。若团队额外想留存完整 CoT 作深度排障，属本地扩展，且必须配套脱敏（见下方 Trace 安全分流）。

**Trace 安全分流（第19章 19.2 合规硬约束）**：Trace 本身是高敏感资产（含明文提示词、业务参数、完整响应），安全要在采集那一刻设计好。采集探针处分流：不含业务语义的元数据（耗时、词元量、状态码）直接进运维仪表盘；含明文提示词、业务参数、完整响应的载荷，先过脱敏管道把可定位到个人的信息替换成不可逆掩码，再加密存进独立权限控制的库。这在合规框架下是硬约束，不是"最佳实践建议"。

（本地实现建议·非书内字面）推荐用 OpenTelemetry（OTel）等标准对接企业现有 APM——书第19章只要求"对接现有 APM / 可观测设施"，未点名 OTel。

## 二、表19-2：运营看板四象限（第19章 19.4 指标经营化）

19.4 的转换问题：信号采上来说的是技术指标语言（首字时延几百毫秒、单次任务三千词元、节点失败率百分之几），按批准键的人关心的是另一种语言（划不划算、风险在哪、钱花在哪）。测量总线的下半程是一道**翻译工序**：把技术指标翻译成经营指标，翻不过去可观测性就进不了决策层视野。

三笔翻译最要紧（19.4）：① **单位任务成本**＝消耗词元 × 模型费率；② **人工接管率换算人力成本**（每降一个百分点估算人力成本变化）；③ **错误恢复率换算韧性**（异常后能否自动恢复）。

翻译完，要有个地方让人三十秒看懂、三分钟定位。书中**表19-2** 把一张好运营看板定为**四个象限**：

| 象限 | 看什么 | 回答的经营问题 |
|------|--------|--------------|
| 左上 · 实时流量 | 当下所有 Agent 的并发量、各步队列深度、吞吐 | 系统现在喘不喘得过气？ |
| 右上 · 成本核算 | 按"业务线→Agent→节点"三级钻取的词元量与货币成本 | 钱具体烧在哪条线、哪个 Agent、哪一步？ |
| 左下 · 质量指标 | 任务成功率、人工接管率、护栏拦截率、错误恢复率的七天滚动趋势 | 它越用越好，还是在悄悄变差？ |
| 右下 · 告警面板 | 按 P0/P1/P2 分级的活跃告警，每条自动关联到对应 Trace | 现在哪里着火了，火源那条 Trace 在哪？ |

> **右下告警面板的关键设计点（19.4）**：每条告警都要**一键跳到它对应的那条 Trace**。看板负责发现"哪里可能有问题"，Trace 负责确认"到底哪一步错了"。少了这根跳转线，值班工程师看到告警还得手动检索执行记录，处置时间全耗在"找证据"上。
>
> **看板的边界提醒（19.4）**：仪表盘是用来发现的，不是用来改进的。汇总数字可能混合信号与噪声，要不要改、改哪里，必须落到 Trace 这个真产物上确认——这直接通向告警判异（见下文第五节"告警是一族控制图"）。

### 注意：这里是【观测面板】，不是【生产运营层的三组看板】

第19章 19.4 的观测面板是上面的**四象限运营看板（表19-2）**，由本 skill（观测层）负责。

子 skill 早期版本里那套"三组看板（车队健康／业务价值／风险）"**不属于第19章 19.4**，它是**第22章 22.6「三组看板：让车队的健康、价值与风险变得可见」（书中表22-4）**的内容，属于**规模化生产运营层**，受众按角色分流（工程运维／CFO／安全合规），由 **`agent-production-gate` 子 skill 负责**，不在本观测层。两者不要混淆：

- **本层（19.4 表19-2）**＝单系统观测面板，四象限，给值班工程师三十秒看懂、三分钟定位。
- **生产运营层（22.6 表22-4，→ `agent-production-gate`）**＝车队级三组看板，按受众分流，各盯一条轴：车队健康盯可靠性（主线 R=∏pᵢ，pᵢ=P(step_i 正确 | 上游状态/上下文/噪声/既有错误)；pⁿ 仅为乐观基线），业务价值盯 ROI，风险盯正交轴 II 与合规底线。

## 三、成本优化（第20章 20.1-20.8 评估与成本工程）

### 不可能三角

```
      质量
      /\
     /  \
    /    \
   /______\
 速度    成本
```

不能同时最大化三者。根据业务优先级选择策略：
- **质量优先**（金融/医疗）→ 接受更高成本和较长延迟
- **速度优先**（客服实时）→ 接受略低质量，用小模型
- **成本优先**（批量处理）→ 接受更长延迟，最大化缓存

### 模型路由规则（第20章 20.4）

书 20.4 给的是**通用档位（厂商中立，不写型号名）**：确定算式归代码（不调模型）→ 有套路的转换 / 抽取 / 填空派**轻量模型** → 多步链式推理 / 全局规划 / 合规终审上**旗舰模型** → 请求入口用**极轻量路由器**实时分发（硬约束：路由器单次判断成本须显著低于其带来的节省额）。典型企业流约六到七成调用是简单任务，把这部分从旗舰降级到轻量，整体模型调用成本通常可观下降。

> **本地实现建议（非书内 · 厂商价格随时变）**：下表把上面的通用档位映射到具体厂商型号与相对成本，属本地落地参考——**第20章未给型号名（Haiku/Sonnet/Opus）与倍数（1x/~4x/~19x）**；用前请按当前厂商定价校准。

| 任务类型 | 推荐模型（本地映射） | 相对成本（本地估算） |
|---------|---------|---------|
| 意图分类/格式化（轻量档） | Haiku | 1x |
| 标准分析/报告生成（轻量~中档） | Sonnet | ~4x |
| 复杂推理/架构决策（旗舰档） | Opus | ~19x |

原则（书 20.4）：**从最便宜的档位开始，只在质量不达标时升级。**

## 四、告警是一族控制图（第19章 19.5）

第19章 19.5 的核心判断：**告警，本质上是一族控制图（Control Chart）。**它的使命是在一条天然有波动的曲线上，分清哪是该出手的信号、哪是不必理会的噪声。

最基本、最有价值的几条告警，其实就是写在控制图上的"判异规则"：

- 人工接管率连续偏高（如持续两小时高于三成）→ 通知工程团队（自主能力出了系统性问题）。
- 单次任务成本越过红线（如一次诊断超预算上限）→ 标记并触发模型路由审查（上下文膨胀或路由失效）。
- 错误恢复失败率走高（如超过两成）→ 熔断并升级（系统恢复能力不足，异常会传到下游）。书 19.5 给的正是这三条"判异规则"示例：接管率连续两小时 >30%、单次成本越预算、错误恢复失败率超过两成（三个阈值均为书内示例值，非本地经验值）；书同时强调阈值不写死、由该信号自身历史控制带派生。

但这些"比如"泄露了真正的难题：**阈值该划在哪？** 第19章 19.5 的答案——一条天然带波动的信号，**单划一根硬阈值线是最糟的做法**：划高了越限时业务已受损（漏报），划低了正常波动频繁越限把人淹没在误报里（最后干脆静音）。正确的不是一根线，而是**一条带**：用信号自己的历史波动算出基线和上下控制限，只有穿出这条带、或出现连续多点同向这种"自然波动几乎不可能造出"的模式，才判一次真异常。

> **Agent 场景的特殊约束（19.5）**：Agent 信号基线本身会随时间漂移（业务模式变、知识库更新、模型版本换）。一根固定硬阈值今天合适、下月就可能多数观测值持续在阈值上方或全程压在下面。基线要变得**版本化、确认后再重估控制限**；没确认前，持续偏移本身就是告警。

**告警这族控制图的真正纪律（19.5 + 本篇原则）**：**不要因单个坏点直接改动系统，先分清这是信号还是噪声。** 对一个只在自然波动的过程频繁微调，会放大它的方差（统计过程控制的经典反 tampering 结论：盯着 X̄-R 图上一个普通下跳点就去调机床，第二天方差比头一天还大）。落到一个坏 Trace 上更是如此：看到一条出错 Trace 就给它打补丁，是对每个坏点动手的极端形式，只会越改越糟。正确做法是**先聚合同类异常 Trace 一并观察**（是不是只有夜班才出？是不是都缺同一个字段？是不是都依赖同一个上游？），当一批坏 Trace 呈现统计上站得住的共同模式时，才是真信号，才值得沿慢外环走整圈根治。

**告警风暴的两个机械解法（19.5，这族控制图的"降噪"逻辑）**：

1. **分级与抑制**：告警分 P0（系统级熔断，立即介入）／P1（Agent 级异常，限时响应）／P2（指标偏移，下个工作日处理），不同级别走不同推送通道。同一 Agent 在同一规则上反复触发，首推之后进入抑制窗口，窗口里只累计计数、不重复推送，窗口结束仍未恢复才推一条聚合摘要。
2. **根因聚合**：多个 Agent 短时间内一起告警时，系统自动查它们是不是共用同一下游依赖。若五个 Agent 两分钟内都报"工具调用失败"且 Trace 显示都挂在同一共享服务上，就聚合成一条"该服务疑似故障"的根因告警，而不是推五条各自为政的告警。

## 五、测量总线如何驱动两环（第19章 19.6）

第19章 19.6 把可观测性还原成一条**双向总线**：向上输入慢外环作为改进的唯一依据，向下回灌快内环做近实时自动调参。对应总论图 0-2 那两条折回的箭头。

### 向上：输入慢外环 DMAIC（书中表19-3 "向上"出口）

DMAIC（界定—测量—分析—改进—控制）每一阶段都依赖这条测量总线提供证据：

- **Define**：是看板上某个信号穿出控制带，才让团队定义出"这里有一个真问题"。
- **Measure**：把相关真实 Trace 一条条拉出来比对，量清问题的真实形状（不信仪表盘汇总，不信子环节转述）。
- **Analyze**：从这批 Trace 的共同模式里识别根因到底在模型、在数据源、还是在某个工具。
- **Improve / Control**：改完之后，这条总线在下一周期说明"该改动是否有效、有没有按反例校验过"。

只放行**穿带的真信号**，沿 D→M→A→I→C 走整圈根治，最后沿 **C→D 判例飞轮**折回起点，把这条教训写进下一版 CTQ 红线（详见第21章 DMAIC 外环·判例飞轮）。

### 向下：回灌快内环（书中表19-3 "向下"出口）

同一批测量回灌快内环做自动调参，最典型是**模型路由**：观测数据反复显示"把格式固定的报表归一化"这种简单步骤用最贵旗舰模型成本收益不匹配，系统据此自动把这一步降级路由给更便宜的轻量模型，单步成本降而质量保持在控制限内。缓存策略、告警阈值也都能被观测数据驱动自我校准。

> **向下回灌的警戒闸门（19.6）**：自动回灌**绝不等于对每个数据点自动反应**。能安全自动调的，只能是那些"穿带信号"已被反复验证、规则清清楚楚的动作（如"这类简单节点一律走轻模型"）；还在自然波动带里跳动的信号，绝不能让系统看见一点抖动就自动改一次参数——那等于把"对噪声做反应"的错误交给机器高速重复。这道闸门就是快内环（毫秒到秒、只对确定信号反应）和慢外环（跨周期、专门分辨信号与噪声）分工的物理边界。

| 方向 | 喂给谁 | 节奏 | 只放行什么 | 对应总论 |
|------|--------|------|-----------|---------|
| 向上 | 慢外环 DMAIC（人＋判例库） | 跨周期（天/周） | 穿带的真信号，沿 D→M→A→I→C 走整圈根治 | C→D 判例飞轮 |
| 向下 | 快内环（路由器/缓存/阈值） | 近实时（毫秒/秒） | 仅已被反复确认的规则化动作，抖动一律挡在闸门外 | 快内环自动调参 |

*（书中表19-3：一条总线、两个出口——向上输入改进、向下回灌调参）*

## 六、数据飞轮的慢外环落地（第21章 DMAIC 外环·判例飞轮）

> 本节是上面 19.6"向上"出口的慢外环延伸。判例飞轮的完整方法论在**第21章**，本 skill 只承接可观测性这一侧的飞轮燃料供给。

```
真信号(穿带) → 拉真实Trace比对 → Trace共同模式定根因 → 一道门禁整批根治 → C→D折回CTQ红线
     ↑                                                                          │
     └──────────────────────────────────────────────────────────────────────────┘
```

慢外环每一圈都基于测量总线输出的真实数据，**不针对每个出错案例分别调提示词**（那正是 19.5 警告的"对每个坏点动手"）——先靠告警这族控制图确认是穿带真信号，再靠 Trace 把根因确认到具体节点，最后只用一道门禁整批根治。

### 评估集保鲜机制（第20章 20.1：量具自己会漂移）

> **本地实现建议**：第20章 20.1 明确评估集本身会过期（量具不校准迟早误导你），要求"保鲜机制"。下列节奏是本 skill 给的具体落地建议，非书内规定的固定周期。

1. 从最近真实业务请求中抽样补充新测试用例
2. 淘汰不再反映当前场景的旧用例
3. **历史反例和边界样本要有固定权重**（第18章：把 Agent 真实犯过的每个错固化成永久反例，测量系统越测越严苛）
4. 评估数据集变更历史纳入版本控制，定期发布评估报告

### 优化优先级排序

> **本地扩展（非书内公式）**：下面这条 ROI 排序公式是本 skill 给的实操工具，书中未给该公式；书第20章 20.6 给的是定性原则"系统的下限由唯一瓶颈决定，先硬化最脆弱、最漏损的那一步，已落进预算的环节别镀金"。

```
ROI = (潜在节省 × 影响任务比例) / 实施成本

优先修复（呼应第20章 20.6 杠杆排序 路由 > 上下文 > 缓存）：
1. 高频+高成本路径（如：Token膨胀严重的报告生成）
2. 高失败率节点（如：特定API调用频繁超时）
3. 高人工接管率场景（如：某类任务判例不足）
```
