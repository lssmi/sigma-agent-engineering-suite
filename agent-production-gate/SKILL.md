## 一、五步生产准入闸门（书22.2 表22-2）

把"上线"从一次主观判断（Demo 跑通+管理层点头）改造成一道客观闸门。Demo 验证的是天花板（最好能多好），生产线要的是地板（最差会多差、多久崩一次）。这道闸门分五步，**由外到内、由轻到重**，每一步都在回答同一问题的一部分：这个 Agent 的可靠性，到底被证明了多少？它是 CDOC（Concept / Design / Optimize / **Capability**）中 **Capability（能力验证）阶段**在"上线"这一刻的展开——不允许上线后再慢慢变好，而是先证明能力、再以灰度把可靠性一点点生长出来。

**铁律：能力门永远排在流量门前面。** 第①~③步不执行任何生产动作，全部用于证明"有没有这个能力"；第④步起才放流量，而流量永远跟在被证明的可靠性后面。把顺序颠倒（先全量放出、出事再补测试）是规模化阶段事故的总源头。

逐步检查，**严格按 22.2 顺序与语义**：

### 闸门①：SOP 就绪性审查（先定验收标准，再进实现，风险敞口=零）
- [ ] 业务部门已交付完整、无歧义的标准作业程序（SOP）：交付什么 / 什么算对 / 什么算错 / 边界在哪 / 哪些动作必须停下询问人工
- [ ] 验收判定器（test oracle）**先于实现确定，且独立于实现**——不允许"等 Agent 写完再回头补定义"（否则是拿被测对象的输出定义被测对象的合格线 = 自证预言）
- [ ] 完成判据、自治边界、人工门触发条件已钉死在墙上

> 回扣主线：oracle 先于实现（CDOC·Capability）。这一步看似"非技术"，却是整道闸门最需严格审查的一关。

### 闸门②：沙盒压力测试（隔离环境里证明能力，风险敞口=零）
在与生产完全隔离的沙盒里接受**三类测试**，专门探查它会以何种方式失败（而非"通常能不能行"）：
- [ ] **功能性测试**：常规场景是否正确（压 `R=∏p_i` 可靠性轴）
- [ ] **对抗性测试**：故意注入恶意指令、塞脏数据，观察能否抵御（压正交轴 II 信任与遏制）
- [ ] **韧性测试**：制造上游超时 / 字段缺失 / 工具失效，观察是优雅降级还是直接崩溃（压 Safety-II"列不出的意外"——退安全停机 / 给带标记的部分结果，还是硬塞陌生输入继续算）

### 闸门③：影子运行（接真实数据、但输出不执行，风险敞口=极低）
- [ ] Agent 接入真实生产数据流，照常感知 / 推理 / 生成结论，但输出**一律不执行，只记录**
- [ ] 运行 **1~2 周** 后，逐条比对 Agent"影子输出"与人类"实际处置"：一致比例多少？不一致处是 Agent 错还是人错？
- [ ] 在零生产风险前提下，取得 Agent 在**真实分布**上的可靠性实测（真实分布恰是沙盒难以完整覆盖的）

> 回扣主线：独立测量、现场看真产物（家族 C）。它本身就是一次低风险的真实测量，把本会在生产暴露的执行失败，提前控制在"不执行"的安全区里。性价比最高的一关。

### 闸门④：灰度放量（流量门跟着可靠性门走，风险敞口=受限/流量封顶）
- [ ] 影子运行通过后才允许真正执行，**起步仅接 5% 真实流量，且仅限低风险顾问型任务**
- [ ] 逐档放量：5% → 20% → 50% → …，**每一档放量都必须先有该档上的可靠性实测作支撑**才允许继续向上
- [ ] **任一档不达标，立即回退到上一档**（5% 出问题回退代价是 5% 而非 100%）

> 回扣主线：可逆放手、能撤的小步走（家族 A 可逆性原则）。灰度曲线本质上就是一条可靠性增长曲线。

### 闸门⑤：全量发布与观察期（持续测量与一键回退就位，全量但可熔断）
- [ ] 流量逐步抬到 100%，**上线后前 30 天明确定义为"观察期"**：监控不放松、人工接管随时待命
- [ ] 一旦实测可靠性**跌破红线**，能一键回退到上一档甚至熔断
- [ ] 独立测量总线接通：每条 trace 被持续采集 / 评估 / 换算成逃逸率，喂给慢外环

> 回扣主线：测量总线接通，慢外环起点。五步准入的终点，恰是持续治理的起点——把 Agent 挂上测量总线，从此持续被测量。

### 本地准入补充检查（融入五步之内、非 22.2 原文五步）

> **本地扩展（非书内）**：以下是更细的可执行检查项，**不是** 22.2 的"第六~第十道闸门"，而是把上面五步落到具体核查动作时的实操补充。按其所压的轴/风险归入对应步骤，不得冒充书内五步顺序。

- **归入闸门②（沙盒压测·正交轴）**：三层护栏已配置并测试（输入/运行/输出）；提示词注入防护已用攻击样本验证；致命三联（读私有数据 + 读不可信内容 + 对外发送）至少拆掉一环
- **归入闸门②（沙盒压测·韧性/Safety-II）**：熔断机制已主动触发验证；工具调用白名单已锁定（default: deny）；上游故障注入下优雅降级而非崩溃
- **归入闸门①/④（SOP·权限边界）**：权限模型矩阵已定义且代码级实现（最小权限）；PII 脱敏管道已部署
- **归入闸门②/④（性能地板，本地阈值建议）**：核心场景评估数据集已建立（建议 ≥50 条）；任务成功率基线已测量（本地建议目标 ≥95%）；P95 延迟已测量且可接受；Token 消耗预算已设定；并发压力测试已通过
- **归入闸门④/⑤（回滚/熔断可逆性）**：所有写入操作支持事务回滚或补偿；模型降级路径已定义（本地路由建议 Opus→Sonnet→Haiku→人工）；回滚触发条件已定义（本地建议：成功率下降 >3% → 限时内回退）；数据库快照/备份策略已配置
- **归入闸门⑤（签字与就绪，对应 22.3 三权分立）**：业务发起人签署上线确认；架构师签署技术可行性；合规官（安全与合规团队·一票否决权）签署准入；运维确认监控就绪；应急预案已制定并演练

### 系统自体 FMEA 登记表（named pattern: Agent-Self FMEA·本地扩展非书内表 / 残余风险台账）

> **本地扩展（非书内表）·准入前必产**：书内 FMEA（terminology B.2 / 第13·17章）是 Safety-I 列**业务**失效模式的方法；这里把它**实例化到 Agent 系统自身**——把散落在各闸门、各子 skill 的自体风险一次聚合成可追溯台账。这是「列得出的失效模式」（家族 B9）在 Agent 自体上的落地，与「列不出的意外只代价一次安全停机」(Safety-II) 互补。severity / occurrence 分级口径为本地建议。

## 二、三权分立与分级审批矩阵（书22.3）

五步准入是一道好闸门，但把守的人不能是被验收的人自己。规模化治理的组织内核是**三权分立**——把一个 Agent 上线的决定权拆给三方各执一把钥匙：

| 角色 | 持有权力 | 职责 |
|------|----------|------|
| 业务部门 | 需求定义权 + 最终验收权 | 判定 Agent 是否满足业务需求和验收标准 |
| 工程团队 | 架构设计权 + 技术实施权 | 决定它如何构建、如何装配控制面 |
| 安全与合规团队 | 准入审查权 + **一票否决权** | 裁定能不能上、风险是否可承受 |

三方权责分立、互相制衡，谁都无法既当运动员又当裁判员。它在数学上对应总论那条不可妥协的原则：**测量必须独立于被测对象**——让构建 Agent 的工程团队同时握有"它能不能上线"的最终裁决权，等于让选手兼任裁判。安全与合规团队那把否决权，就是**组织级的独立量具**，它的职责不是配合上线，而是专门去寻找"这个 Agent 还不该上"的反例（家族 C：测量的人不能是被测量的人，裁判要去找反例，不是收集对勾）。

**分级审批矩阵**——按 Agent 业务风险等级配不同放行通道（总论"控制档位要匹配任务风险与能力差异"在治理层的复现）：

| 业务风险等级 | 放行通道 | 说明 |
|------|----------|------|
| 低 | 快速通道，两周内放行 | 只读内部知识问答类，出错可撤回 |
| 中 | 简化五步（可压缩沙盒/影子周期） | 受限执行、影响可控 |
| 高 | 完整五步准入 | 会动手执行、影响较大 |
| 关键 | 完整五步，一关不省 | 会触及不可逆动作（如下令产线全面停机、向供应商发召回公函） |

> 给低风险 Agent 套全套审批 = 浪费治理预算、还会把团队逼向非合规渠道；给关键风险 Agent 走快速通道 = 把关键业务风险暴露给未经验证的放行决策。**控制档位，至少要和任务风险一样多。**

## 三、版本治理 SOP（本地运维扩展，非22.2）

> **本地扩展（非书内）**：22.2 的原文是"五步生产准入闸门"，**不包含**下面这张"版本治理 7 项检查表"。本表是把书 22.5 持续改进飞轮、22.6 看板复审、22.3 三权分立等精神落到**日常版本变更**的本地运维 SOP，**不冒充 22.2 来源**。书内对应锚点：22.5（飞轮只对趋势响应、每次根因修复固化为下一版准入红线与判例库）、22.4 表22-3 第9条（季度复审护栏有效性 / 查判例库过期 / 复核权限范围）。

每次版本变更逐项核查，每季度全量审计：

| # | 检查项 | 频率 | 责任人 | 书内锚点 |
|---|--------|------|--------|----------|
| 1 | Prompt 版本纳入 Git 版本控制 | 每次变更 | 工程+业务联合签字 | 22.3 三权 |
| 2 | 知识库更新触发回归测试（核心场景） | 每次知识变更 | 知识管理团队 | 22.5 飞轮 |
| 3 | 模型升级执行灰度发布+回滚（5%→逐档，不达标即回退） | 每次模型变更 | 工程团队 | 22.2 闸门④ |
| 4 | 护栏规则季度审计（触发次数/有效率/误报率） | 每季度 | 安全+业务 | 表22-3 第9条 |
| 5 | MCP 服务器/工具变更触发兼容性 + 渗透/代码审计 | 每次工具变更 | 工程+安全 | 表22-3 第3条 |
| 6 | 判例库季度抽检+过期淘汰 | 每季度 | 领域专家 | 表22-3 第9条 / 22.5 |
| 7 | 评估数据集月度刷新 | 每月 | 质量团队 | 22.5 飞轮 |

> **飞轮纪律（书22.5）**：版本变更的依据是**趋势，不是单个坏 trace**。看到一次错就冲上去改 prompt / 调阈值，是给本来稳定的过程做无依据调整（tampering, 休哈特"不要对噪声做反应"），越改越糟。慢外环之所以"慢"——先攒够数据、分清信号与噪声，再改、改完固化为下一版准入红线。

## 四、中国合规语境下的上线治理清单（书22.4 表22-3）

三权分立解决"谁来把关"，把关标准里有一组在中国部署 Agent 时绕不开的硬约束。核心工程动作是把**模型调用层抽象成可插拔接口**（总论"把会变的和不变的解耦"原则：底层模型从境外 API 切换为国产私有化部署时，只换适配器，上层护栏/状态机/审批网关一行不改）。

下面是**表22-3 的 10 项治理动作 × 架构落点**（书内原文逐条，建议与法务/安全/IT 治理团队共核；具体法律义务以企业法务和最新监管要求为准）：

| # | 治理动作 | 架构落点 | 状态 |
|---|----------|----------|------|
| 1 | 模型接入审批：公众服务 Agent 完成算法备案，内部 Agent 过企业安全委员会审批 | 统一接入网关 | □ |
| 2 | 数据出境评估：调境外 API 前确认是否涉个人信息/重要数据出境，涉及则走安全评估 | 可插拔模型层 + 脱敏管道 | □ |
| 3 | 工具注册审批：每个工具/MCP 服务上线前过安全团队的渗透测试与代码审计 | 工具注册中心 | □ |
| 4 | 日志留存策略：运营级 trace 留 90 天，涉人工审批的 trace 留 3 年（满足审计追溯） | 测量总线 + 决策台账 | □ |
| 5 | 敏感数据脱敏：流经上下文的个人信息经自动化 PII 脱敏，支持"被遗忘权"执行 | 网关侧脱敏中间件 | □ |
| 6 | 外发动作白名单：发邮件/短信/对外 API 等外发动作纳入白名单，非白名单网关拦截 | 最小权限 + 网关机械执行 | □ |
| 7 | 内容安全过滤：面向终端用户的输出经内容安全模块检测后才放行 | 输出侧护栏 | □ |
| 8 | 上线前红队测试：对 Agent 做对抗性提示词注入测试，覆盖主流注入威胁清单前几项 | 沙盒压测（正交轴） | □ |
| 9 | 季度复审机制：每季度审护栏有效性、查判例库过期、复核权限范围 | 慢外环 + 护栏审计 | □ |
| 10 | 应急预案：每个生产级 Agent 配"一键熔断"手册，每季度做一次熔断演练 | 封顶/安全停机 | □ |

> **第 6 条尤其要害**——总论正交轴 II 在合规层的硬落地：外发约束**绝不依赖在提示词里恳求模型"不要乱发"**，而是在网关上把非白名单动作机械拦下。哪怕模型被藏在上传报表里的注入诱导得想对外发通知，网关白名单里根本没有那个目标地址，它想发也发不出去。合规不是写在文档里的承诺，而是固化在网关上的一道代码闸。

## 五、影子 AI 收编（书22.1 表22-1）

规模化之前先承认一个事实：企业内部很可能已有几十个"无人管理的 Agent"在运行，只是尚未进入统一台账与审计视野——这就是**影子 AI（Shadow AI）**：业务部门绕过 IT 治理，私自把大模型接进工作流并未经治理地输入真实业务数据。

它是规模化头号敌人，因为它**既不可靠（无三层控制面、p 抬不起来 / N 缩不下去 / 失控封不了顶）、又不设防（无定界包裹、信任轴形同虚设）、还查无此人（从未接入测量总线，事故无法溯源）**。

**正确路径是收编而非封杀**：一刀切禁用在车间几乎无法落实，封堵越严绕行越多。建一个安全且好用的官方平台，让走合规渠道比绕行更顺手，影子 AI 才会自行退潮。关键是先把"合规使用"这项**基本期望项（must-be requirement, A4）做扎实**——官方平台首先要做到的不是更惊艳，而是使用成本更低、可信度更高、误拦截更少（审批等两周/卡顿/拦错合法请求，员工照样绕回非合规渠道）。

表22-1 从影子 AI 到官方平台收编（把"查无此人"变成"全程可测"）：

| 维度 | 影子 AI（私自使用） | 官方平台收编后 |
|------|---------------------|----------------|
| 可见性 | 谁在用/用什么/喂了什么数据，全不可知 | 统一网关，每次调用可计量、可审计 |
| 可靠性（R=∏p_i） | 裸模型直接干活，无控制面，无封顶 | 纳入三层控制面，按任务分诊配档 |
| 安全（正交轴） | 外部内容直接入模型，注入门户大开 | 定界包裹 + 最小权限 + 网关机械执行 |
| 合规证据 | 查无此人，事故无法追溯 | 结构化轨迹与决策台账，可回溯举证 |
| 对应原则 | — | A4 先把 must-be 做到位（让正门比偏门省心） |

上线前确认企业影子 AI 治理就位：
- [ ] 已摸底企业内部使用的未审批 AI 工具数量与数据流向
- [ ] 已建立或计划建立统一接入网关（每次调用可计量、可审计）
- [ ] 官方平台体验已达"比私接更省心"（A4 must-be 先到位）
- [ ] 已建立业务/工程/安全合规"三权分立"治理架构（见第二节）
