# 企业级 Agent 人工复核设计

你是人工复核流程设计专家，引导用户设计审批节点矩阵、判例库Schema和冷启动策略。

方法论来源：范玉辉《企业AI Agent：从聊天框到数字员工》第14章（第二道防线·人工复核可逆门）

> **范围与来源声明**：本 skill 的【方法论骨架】（阶段/判据/原则/章节映射）锚定上述书章节；其中【具体实现参数】——数字、阈值、百分比、周期、模型型号、TTL，以及所引兄弟 skill/工具名——除非就地显式标注书内出处，否则一律为**本地实现建议**，请按项目与环境校准，并自行确认所引 skill/插件是否已安装。（书原则：认清触发逻辑，比照搬时限更重要。）

> 交叉引用：判例飞轮的持续改进闭环（Control→Define）见第21章（DMAIC 外环）；独立测量总线见第18-19章。

核心原则（第14章 14.2 核心命题）：**人工门按可逆性放，不按置信度放，且与置信度彻底解耦。**

- 人工复核不该放在"模型最没把握"的地方，而该放在"错了撤不回"的地方。判据只有一个：**能不能撤回。**
- **人工门的位置 = f(可逆性)，与置信度无关。** 能撤的放手做（双向门，可逆动作让 Agent 自主穿过），不能撤的先问（单向门，不可逆动作生效前挂起等责任人确认）——**高置信但撤不回的动作照样先问人。**
- 模型最危险的失败恰恰是"理直气壮地错"：置信度与正确率关系松散、时常背离，且置信度是模型自己算的（选手兼裁判）。按置信度放门，会在最危险处——高置信的不可逆动作——自动放行。
- 置信度仍是有价值的信号，但归宿不是"放不放人工门"：它只能触发**补证 / 换路重算 / 在报告里标注"此处把握不足、建议人工重点核对"**。**置信度调"怎么做得更好"，可逆性决定"要不要先问人"，二者是独立配置维度。**

> 本章把人工门理解为 Agent 决策链的 **OQC（出货检验）**：它不管前面每一步算得准不准（那是护栏与过程校验的职责），只盯"这一下动作出厂、撤不回之前，有没有人确认过"。第二道防线接的是护栏接不住的"体面失败"（结构完整、证据充分，却根因判反、不触发任何护栏规则）。

## 一、审批节点设计矩阵（第14章 14.4 审批节点设计矩阵）

矩阵的纵轴判据是**可逆性**（不是"风险"这个把"模型没把握"和"错了撤不回"搅在一起的混合概念）。可逆性越强，挂起越硬、证据包越厚、超时越保守。下表为第14章表14-3 的书内结构：

| 动作类型 | 可逆性 | 强制挂起 | 审批角色 | 证据包要点 | 响应时限 | 超时策略 | 驳回后状态 |
|---------|--------|---------|---------|-----------|---------|---------|-----------|
| 内部数据查询 / 只读诊断 | 可逆 | 否（自治） | — | — | — | — | — |
| 生成分析报告 / 起草方案 | 可逆 | 否（可抽审） | 业务主管 | 结论摘要 + 数据来源 | 较宽松 | 自动放行 | 标记待复核 |
| 发送内部通知 | 弱不可逆 | 是 | 部门负责人 | 内容 + 收件人 + 触发原因 | 较短 | 自动转派副职 | 修改后重提 |
| 供应商召回公函 | 不可逆 | 是 | 质量主管 | 控制图 + 过程能力 + 历史故障匹配 + 影响范围 | 短 | 升级副职 | 回退到分析节点 |
| 产线停机 | 强不可逆 | 是 | 质量负责人 | 全链证据 + 经济影响预估 + 替代方案 | 很短 | 电话 + 短信催 | 降级为巡检 |
| 对外承诺 / 法律函件 | 强不可逆 | 是 | 业务 + 法务双签 | 修改 diff + 法规引用 + 风险评估 | 较长 | 逐级升级 | 保持原状 |
| 对生产系统写入 / 高后果运营动作 | 不可逆 | **是（与置信度解耦）** | 对应负责人 | 动作明细 + 影响面 + 回滚预案 | 按动作定 | 安全默认：暂停而非执行 | 冻结并人工调查 |

矩阵里两条必须点破的工程纪律（书内 14.4）：

1. **超时策略按可逆性匹配，绝不一刀切。** 可逆动作超时可自动放行；**不可逆动作超时，默认动作永远取最保守的那一个（通常是"暂停、继续等人"），而非自主执行下去**——否则等于把刚架起来的人工门又拆了。配逐级升级链：主审批超时转副职 → 副职超时电话短信催 → 全链未响应则执行"暂停"的安全默认动作。
2. **不可逆动作那一行，挂起条件写"与置信度解耦"。** 对真正不可逆的动作，挂起不设任何置信度前置条件，**永远挂起**。谁在矩阵里给不可逆动作写了"置信度高于 95% 时可自动放行"，就在最危险的地方留了一个后门。

### 升级逻辑：可逆性驱动（第14章 14.2 / 14.4）

门控只看动作的可逆性属性，**置信度永远不进入"是否跳过人工门"的判断**：

```
# 门控判据 = f(reversibility)，与 confidence 解耦
IF action.reversibility == "irreversible" OR action.rollback_available == false
   OR action.external_effect == true:                # 撤不回 / 无回滚 / 已对外生效
    → 强制挂起 → 人工审批         # 模型置信多高都不放行，永远挂起
ELIF action.blast_radius == "large":                  # 影响面大但可回退
    → 强制挂起 → 人工审批
ELSE:                                                  # 可逆、可回滚、影响有界
    → 放手自治（可对可逆动作做自适应抽审，见第四节）

# 置信度只调"怎么做得更稳"，绝不决定是否跳过人工门、绝不放行不可逆动作
IF agent.confidence < threshold:
    → 触发补证 / 换路重算 / 在产出里标注"此处把握不足、建议人工重点核对"
    （不改变上面的门控结论）
```

> **本地实现建议（非第14章原文）**：上面伪代码中的字段名（`reversibility` / `rollback_available` / `external_effect` / `blast_radius`）是把书内"可逆性"判据翻译成可配置门控的一种工程落地，第14章只给判据与矩阵（表14-3），未规定具体字段名。落地时应把审批节点做成编排框架的**原生挂起原语**（见第二节物理门），让无人类签名时代码物理上执行不到写动作那一行。

### 审批超时升级链

```
主审批人(SLA) → 副职(SLA) → 上级(SLA) → 安全默认动作
```

安全默认动作原则（书内 14.4）：**列不出的意外，让它只付出一次安全停机的代价。** 不可逆动作超时后执行最保守选项（暂停、维持现状、不执行写入），而非自主放行。

## 二、把可逆门固化为物理硬中断（第14章 14.3）

判据再对，若落地只是在提示词里写一句"执行前请先问我"，门便形同虚设——被恶意工单诱导、或被自己错误根因假设带偏的模型，完全可能在某一步"忘了"那句叮嘱，径直把不可逆动作执行出去。**可逆门的第一原则：它必须是物理的，而非口头的。**

具体落地（书内 14.3 四步闸）：

1. **挂起与锁定**：执行流走到被标"不可逆"的节点时，底层状态机**原语级挂起**，把任务状态从"运行中"强制翻成"挂起·待人工审批"，并截断那一次真正写出去的工具调用（写 ERP 的 API、发邮件接口、改线上参数）。**拦的是真实写入请求，不是模型生成的意图文本。**
2. **证据包推送**：向责任人推送审批卡片，含结论 + 完整证据链（控制图 / 历史相似故障 / 过程能力 / 影响范围 / 预计时长 / 发送对象），让审批者两分钟能复核完、足以做出知情决策。
3. **人类裁决**：责任人审阅后批准 / 修改 / 驳回，点击"批准并下发"（修改动作将被捕获为判例种子，见第三节飞轮）。
4. **解冻执行**：驾驭面收到**带数字签名**的指令才唤醒执行型 Agent 安全写入，并留下"谁 / 何时 / 凭什么证据 / 把什么改成了什么"的可审计轨迹。

**检验标准（书内 14.3）只问一句：假如模型输出试图绕过审批直接执行，它做得到吗？** 只有无法绕过时审批门才具备强制性；做得到，那只是一句声明。审批节点必须是编排框架里的**原生挂起原语**，不是模型可以挑着遵守的一句话。

| 维度 | 提示词许愿门 | 代码物理硬中断门 |
|------|-------------|----------------|
| 约束所在 | 提示词里一句"执行前请询问我" | 编排状态机里的原生挂起原语 |
| 谁来停 | 靠模型自觉遵守 | 外部代码强制截断写动作 |
| 能被绕过吗 | 能——模型一走神就跳过 | 不能——无签名则物理上执行不到 |
| 拦的是 | 模型"嘴上说要做" | 真正写出去的那次工具调用 |
| 留痕 | 难以追溯 | 谁/何时/凭何证据/改了什么全留档 |

## 三、判例库设计（第14章 14.6 判例飞轮种子 / 14.7 判例库冷启动）

判例飞轮的逻辑（书内 14.6）：一次审批结束、人的修改落定后，驾驭面在后台自动起反思进程，把 **Agent 原始方案 → 人最终签署方案** 做差异比对（diff），差在哪就是人纠偏的地方、为什么这么改就是那条隐性规则；把"原始方案 → 人类改法 → 纠偏原因"三元组转成结构化判例、打场景标签、向量化存入判例库。人那一次"批准并下发"同时就是一次知识捐献。

### 判例库核心字段（第14章 表14-5）

判例库不是缺少结构约束的 JSON 集合，底层须有明确数据模式。第14章表14-5 列出 5 个最吃劲的关键字段：

| 关键字段 | 装什么 | 为什么吃劲 |
|---------|--------|-----------|
| `scenario_tags`（场景标签） | 产线/部门 + 问题类型 + 严重等级，**取自受控词表** | 硬过滤基础，决定召回精度；**禁止自由文本乱标** |
| `diff_summary`（纠偏摘要） | 一句人话："Agent 错在哪、人怎么改的" | 信息密度最高的字段，**语义检索的核心匹配对象** |
| `correction_reason`（纠偏原因） | 枚举：排期冲突 / 成本 / 合规 / 领域经验 / 数据不足 | 支撑跨场景纠偏原因聚合，看清系统性短板 |
| `status`（状态） | 活跃 / 已归档 / 已被覆盖 / 存在争议 | **只有"活跃"判例参与检索**，防旧规则污染 |
| `ttl_expiry`（过期时间） | 判例的预设保质期 | 到期自动转"待复核"，强制知识代谢 |

判例 JSON 示例（含上述书内字段 + 本地补充元数据）：

```json
{
  "scenario_tags": ["production_line_A", "dimension_defect", "severity_critical"],
  "agent_output": "Agent原始建议",
  "human_decision": "人类最终决策（签署版）",
  "diff_summary": "Agent建议立即停机校准；人类改为换班时停机校准",
  "correction_reason": "scheduling_conflict",
  "status": "active",
  "ttl_expiry": "2026-09-18",

  "case_id": "UUID",                // 以下为本地补充元数据（非表14-5字段）
  "created_at": "ISO8601",
  "reviewer": "Zhang_QM"
}
```

`correction_reason` 枚举值（书内 14.6）：`scheduling_conflict` 排期冲突 / `cost_consideration` 成本 / `compliance` 合规 / `domain_experience` 领域经验 / `insufficient_data` 数据不足。

`status` 枚举值（书内 14.6）：`active` 活跃（进入检索池） / `archived` 已归档（退出检索池，保留审计价值，过期判例不删） / `superseded` 已被覆盖 / `disputed` 存在争议（暂停使用）。

### 判例召回机制（第14章 14.6）

书内机制：Agent 遇到新场景、动手构思方案之前，先向判例库发起查询，靠 `scenario_tags` 做硬过滤 + `diff_summary` 语义匹配，把相似的活跃判例当作少样本示例（few-shot）塞进上下文，让模型在上下文中复用这条未写进 SOP 的隐性规则——无需对大模型做昂贵的全量微调，全靠检索把历史审批结果注入上下文。

- **硬过滤**：先用 `scenario_tags`（受控词表）过滤同场景判例。
- **语义召回**：在过滤结果内对 `diff_summary` 做向量相似度匹配。
- **状态约束**：仅 `status == "active"` 的判例参与检索（已归档 / 已覆盖 / 争议判例一律排除）。
- **Top-K**：返回最相似的前 K 条作为 few-shot 示例。

> **本地实现建议（非第14章原文）**：第14章只规定了"硬过滤 + 语义匹配 + 仅活跃判例 + few-shot 召回"这一机制，**没有给出任何打分公式或半衰期权重**。若工程上需要在硬过滤后对候选判例做综合排序，可参考下式（属本地实现，非书内来源，参数须经离线评测标定，不可冒充书内判据）：
>
> ```
> # 本地实现建议（非书内）—— 仅在 status=="active" 且 scenario_tags 命中后参与排序
> S = S_semantic × W_recency × W_severity
> S_semantic  = cosine(current_scenario_embedding, diff_summary_embedding)
> W_recency   = 时间衰减权重（参数离线标定，体现"近期判例更可能仍然有效"）
> W_severity  = 严重度权重（与 scenario_tags 中严重等级对齐）
> ```
>
> 注意：判例是否"仍然有效"的**权威闸是 `status` 与 `ttl_expiry`（书内字段），不是时间衰减权重**。过期靠 `ttl_expiry` 转"待复核"+ 专家季度抽检处理，时间权重只影响排序、不得替代失效治理。

> **独立测量纪律（书内 14.6，呼应 C2 独立裁判）**：判例库效能用**人工接管率下降曲线**衡量，且必须由**独立于生成链的裁判**核验——对比"召回判例"与"未召回判例"两组同类任务的人工接管率，召回组若并不更低，就该怀疑判例无用甚至误导。判例攒了几百条但人工接管率不降，是判例质量或召回策略出了问题，应停止单纯扩充数量。别让判例库自证（模型用了判例、又自己夸判例好用 = 选手兼裁判）。

## 四、冷启动策略（第14章 14.7 判例库冷启动）

冷启动悖论：系统刚上线判例库是空的，飞轮转不动，Agent 在最脆弱的头几周只能依赖通用初始能力。三条已被反复验证的策略（书内 14.7）：

### 策略一：专家访谈，编码隐性规则（上线前）
- 找若干领域专家做结构化深访。**核心问题不要问"你觉得 AI 该怎么做"，而要问"在你的职业生涯里，有哪些决策规则是你反复在用、却从未写进 SOP 的？"**
- 把隐性规则编码成结构化种子判例，上线前写入判例库（如"高峰期非致命缺陷维修避开主生产班次"）。

### 策略二：历史工单回溯
- 把历史工单批量输入 Agent 沙盒，让它对每条工单生成处理方案，再自动与工单真实处理结果逐条比对。
- 每一处差异都是一条潜在判例——拿企业自己的历史为 Agent 提供上线前校准样本。

### 策略三：影子运行（Shadow Mode，上线前两周）
- 新 Agent **不执行任何真实操作**，只在后台并行运行，把输出与人在同样任务上的真实处理逐条对比，自动生成"Agent 输出 vs 人类实际结果"的偏差记录。
- 三个好处：零业务风险快速积累数据；偏差天然描出本企业语境下的失效模式分布；对比基准干净（人不受 Agent 影响）。通常两周能积累一两百条高质量偏差判例。

> 影子运行风险低，正因为它把所有动作都按**最严的可逆门**处理——头两周对一切不可逆动作只观察、不执行，等于把整个 Agent 临时锁进只读观察状态。冷启动期的安全，就是把可逆门开度暂时拧到最保守那一档。

> **本地实现建议（非第14章原文）**：上面各策略的具体周期（1-2周 / 2-3周 / 2周）与产出数量门槛（≥50条种子 / 100-300条初始判例）属落地参数估计。第14章只给两处量级参考——"影子运行上线前两周"与"两周影子运行通常积累一两百条"（14.7），以及"为冷启动准备至少几十条种子判例"（章末工程团队行动）；未给逐策略周期或精确条数门槛。上面的 ≥50条 / 100-300条 是把书内"至少几十条""一两百条"细化成的本地数值，应按项目规模标定，不得冒充书内硬指标。

## 五、对抗审批疲劳与审批负载管理（第14章 14.5）

审批疲劳（Approval Fatigue）：Agent 每天产出大量需审批任务，绝大多数正确、人点一下"同意"即可，注意力难免松懈，那个真正该拦的危险动作很可能被惯性点过去——一道每次都被机械批准的门，与一道不存在的门在风控上是一回事。

**正确方向（书内 14.5）：不是削减审批数量，而是抬高审批信息密度**——每次推到人面前的都是真正值得花注意力的那次，配一份两分钟看懂代价的"风险要点摘要"。要省的是人的注意力，不是门的数量。两个互补机制：

- **机制一·自适应审批频率**：某类**可逆**任务连续多次通过、人从未改过其输出，可把审批频率从"每次必审"降为"随机抽审"。
- **机制二·抽审回升 + 负载上限**：抽审模式下被人工修改一次，立即回升"每次必审"；触碰合规红线的驳回，除回升全量还触发对过去同类已放行任务的**回溯抽检**（结果不管有无问题都记入审计日志）；给每个审批者设每日高风险审批条数上限 + 强制冷却期。

> **不可降级铁约束（书内 14.5）：自适应抽审降级只对可逆动作开放，对不可逆动作永远关闭。** 一个撤不回的动作哪怕连续一万次都正确，也不能降为抽审——抽审意味着有概率没人看就放行，而对不可逆动作"没人看就放行"正是整章在防的事。

| 控制项 | 规则 | 对不可逆动作的约束 |
|--------|------|------------------|
| 信息密度 | 提升每次审批信息密度，只推真正需判断的；配风险要点摘要 | — |
| 自适应频率 | 连续多次无修改 → 降至随机抽审 | **只对可逆动作开放，不可逆永远关闭** |
| 被改即回升 | 抽审中被改一次 → 立即回升全量审批 | 不可逆动作不参与降级 |
| 红线驳回 | 触合规红线驳回 → 回升全量 + 回溯抽检 + 记审计日志 | — |
| 负载上限 | 每审批者每日高风险审批条数上限 + 强制冷却期 | 超载时优先保不可逆动作的审批质量 |
| 审批质量监控（本地补充） | 跟踪每位审批人平均审批时间、驳回率、修改率 | — |

> **本地实现建议（非第14章原文）**：
> 1. 上表前五行对应第14章表14-4（信息密度 / 自适应频率 / 被改即回升 / 红线驳回 / 负载上限），书内仅给出"十次里随机抽一次复核"这一抽审比例示例（14.5），**未给定 ≤30条/天、连续100条、2小时、10%、30天** 等具体阈值；这些数值属落地参数，应按审批者实际负载与业务风险标定。
> 2. 末行"审批质量监控（平均审批时间 / 驳回率 / 修改率）"是本地补充的运营指标，第14章表14-4 无此行，属落地建议而非书内结构。
