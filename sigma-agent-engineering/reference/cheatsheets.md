# 速查表（Cheatsheets）

> 本文件是 sigma-agent-engineering 主 skill 的随手可查速查层，对应专著《企业 AI Agent：从聊天框到数字员工》(Sigma Engineering 范式) 附录 C / D / F。
>
> 内容编排：
> - **C 段** — 22 章逐章一句话核心问题（上线自检单），按三篇分组。
> - **D 段** — 阶段 × 仪器 × 交付物总表（DMAIC 五阶段 + CDOC 四阶段）。
> - **F 段** — 主干图索引（论证图 + 导览图的必查主干）。
> - **附** — 成熟度快查法。
>
> 主线钉死：任务整体可靠度 **R ≈ pⁿ**，p = 单步可靠度、N = 任务步数 (horizon)。真实写法是条件概率连乘 **R = ∏ pᵢ**（pᵢ = P(step 正确 | 上游状态、上下文长度、噪声密度、既有错误)）；**pⁿ 只是便于建立直觉的乐观基线**，真实衰减因 self-conditioning 误差传播更陡，不可把它简化成纯 pⁿ。0.95²⁰ ≈ 0.36。
>
> 建造期路线 **CDOC = Concept–Design–Optimize–Capability**，末阶段是 **Capability（能力验证）**，不是 Control；DMAIC 的末阶段才是 Control。两者不可混。

## D　阶段 × 仪器 × 交付物总表

来源：附录 D「阶段 × 仪器 × 交付物总表」(file28)。用法（附录 D 原文）：先看你在哪一行（阶段），再看该挂哪些仪器（中间列），最后核对你有没有产出该阶段交付物（末列）。**把表贴墙、让每个人指出项目站在哪一行、把该行末列交付物一件件摆到桌上——摆不出来的那几件，就是真问题所在。**

### D.1　慢外环：DMAIC 五阶段（改进既有 Agent）

| 阶段 | 这一步在干什么 | 主挂仪器 | 典型交付物 |
|---|---|---|---|
| **D 定义 Define** | 挑对要解的问题、定 CTQ、给任务分诊、定控制档位 | JTBD / 设计思维（要结果）· Cynefin + 必要多样性（按问题类型分流）· 单向/双向门（按可逆性放人工门）· Kano（must-be 先到位） | 问题陈述与 CTQ 清单 · 任务分诊表（问题类型 × 可逆性 → 控制档位）· 人工门放置清单 |
| **M 测量 Measure** | 拉真实 trace、建独立量具、把表现换算成逃逸率 | DMAIC-Measure · MSA / GR&R（量具独立性）· 现地现物（看真产物，不信仪表盘 / 子代理转述）· 过程能力 Cp/Cpk | 独立测量系统设计 · trace 采集规范 · 逃逸率 (escape-ppm) 基线 · 失败样本集 |
| **A 分析 Analyze** | 找主导因子与瓶颈步，区分信号与噪声、事实与假设 | TOC（定位最低 p 的瓶颈步）· 波普尔证伪（去找反例）· 休哈特/戴明（先分信号噪声，不 tampering）· 共因失效分析（冗余是否会一起错） | 根因分析报告（标注事实/推断/假设）· 瓶颈步定位 · 信号 vs 噪声判定 |
| **I 改进 Improve** | 用最小有效改动解决根因，三手段上手 | 精益（↓N，删随机步、做减法）· 防错 / DFSS（让错的动作做不出）· 田口稳健（对噪声不敏感）· TRIZ（化解矛盾不折中）· 误差预算（劲使到真瓶颈） | 改进方案（标注手段 ①/②/③）· 新增/收紧的护栏 · 缩短后的链条 N · 稳健性验证结果 |
| **C 控制 Control** | 把修复焊成永久门禁，沿 C→D 飞轮折回 Define | PDCA / 判例飞轮（教训焊成永久门）· 戴明 94/6（修系统不骂模型）· 验证者定律（把预算花在独立检查者上）· SRE 监控 | 固化的快内环新门禁 · 判例库新条目 · 下一版 CTQ 红线 · 看板指标定义 |

### D.2　建造期：CDOC 四阶段（设计全新 Agent）

| 阶段 | 这一步在干什么 | 主挂仪器 | 典型交付物 |
|---|---|---|---|
| **C 概念定义 Concept** | 识别机会、约束、用户真实任务与成功指标 | JTBD（用户雇佣的是结果）· Kano（区分 must-be / 期望 / 兴奋）· Cynefin（问题落在哪个域）· 设计思维 | 机会与约束清单 · 成功指标 (CTQ) · must-be 基本功界定 · 替代方案对比 |
| **D 设计 Design** | 先定接口、数据契约、状态机边界、验收 oracle，再实现 | DFSS（质量设计进去）· 防错 / Poka-Yoke（让错做不出）· 三层控制面 · 状态机 / 编排 · 状态外置 + 幂等（封顶前置） | 接口与数据契约 · 状态机定义 · 验收 oracle（独立于实现）· 三层控制面骨架 · Schema 校验规则 |
| **O 优化 Optimize** | 做权衡、跑实验、把可靠性曲线压到目标落点 | 田口稳健设计（对噪声不敏感）· TRIZ（解自治 vs 可验证的矛盾）· TOC（硬化瓶颈步）· 误差预算（不镀金、对准瓶颈）· 精益（↓N） | 权衡矩阵 · 稳健性 / 抗噪实验结果 · 瓶颈硬化方案 · 误差预算分配表 |
| **C 能力验证 Capability** | 用真实输入、独立 oracle、回归集证明上线前已具能力 | DFSS-Capability（上线前证能力门）· 可靠性工程（FMEA / Safety-I + Safety-II 验证）· MSA（验收量具独立）· 灰度 / 影子运行 | 上线前能力门报告 · FMEA 与降级策略验证 · 回归集与逃逸率验收 · 五步准入记录 |

> **跨阶段复用（附录 D 原文）**：同一门方法论在不同阶段反复出场是正常的——TOC 在 Analyze 定位瓶颈、在 Optimize 硬化瓶颈；田口稳健既在 Improve 也在 Optimize。表中「主挂」只列该阶段最吃重的几门，并非排他。所有仪器在 pⁿ 上的手段归属见附录 E 与总论表 0-1。
>
> **CDOC 末阶段务必是 Capability（能力验证），不是 Control。** DMAIC 的末阶段才是 Control。两条路线的末阶段不同，是本书反复钉死的诚实性约束。

## 附　成熟度快查法（缺仪器即前兆）

来源：附录 G 决策者要点 + 附录 D 说明（file28）。

**快办法**：拿附录 D（D 段表）去对照，看项目现在走到哪个阶段、该阶段「典型交付物」那一列缺了哪几样。你声称走到哪个阶段，就把该阶段交付物逐项摆出来核——摆不出来的那几件就是真问题。

**两个红旗判据（Demo 能跑、生产会崩的前兆）：**

1. **缺 Measure 阶段的独立量具** —— 没有独立于被测对象的测量系统（让 Agent 给自己打分 = 自证清白），慢外环失去唯一计分依据，只能凭主观改动。对应自检问题：第 3 章 / 第 18 章「测量系统独立于被测对象吗？」
2. **缺 Capability 阶段的上线前能力门** —— 跳过 CDOC 的 Capability 直接放行（号称「已经到 Capability」，一问上线前的独立能力门报告没有、再问 FMEA 与降级策略验证也没有）。这不是到了 Capability，而是**跳过了 Capability 直接放行**。对应自检问题：第 11 章「上线前证过能力没有？」

> 附录 D 原文：「那不是到了 Capability，而是跳过了 Capability 直接放行。这张表的作用，正是把项目阶段的自我判断与实际交付物之间的落差暴露出来。」

**带不走的组织资产**（附录 G 决策者要点）：真正可沉淀的资产集中在附录 C 与 附录 E —— 一套由各章 Checklist 固化下来的工程纪律（C 段），和一套有明确方法论来源、可被任何接班人读懂并接续的判例库（方法论出处账本，见 methodology-ledger）。**模型会换，这两样不会。**

## 附2　Sigma 工程成熟度矩阵（named pattern: Sigma Maturity Matrix）

> **本地扩展（非书内表）**：把上面的「成熟度快查法」（散文）升级为可勾选网格——评的是 **Agent 工程系统自身**装到了哪一档（**不是**流程成熟度 L1-L5，那是 `enterprise-agent-assessment` 评业务流程的）。行=书内既有工程维度（锚 methodology-ledger 家族 + 三道防线 + 测量总线），列=四档；四档定义为本地口径。

四档（每格按**可观察证据**判，不靠主观）：**0 缺席**（没有）/ **1 文档级**（只写了要求、无机械 enforce）/ **2 机械 enforce**（有 gate/hook/schema/代码门，但未跑真样本）/ **3 闭环**（机械门 + 真样本验证 + 失败固化回判例库）。
