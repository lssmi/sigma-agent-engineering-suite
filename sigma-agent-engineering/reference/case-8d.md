# 贯穿案例（一）：制造业 8D 诊断 Agent 端到端走完整圈

> 本文是主 skill `sigma-agent-engineering` 的两条贯穿案例之一（另一条见电商只读经营决策平台）。它沿 P0–P11 十二阶段流水线把一个制造业 8D 诊断 Agent 从立项一路走到生产准入，每一阶段标出本案例的落点，并把全书 R≈pⁿ 主线、两速控制系统、三层控制面、三道防线、六方法论在同一个主角身上兑现一遍。
>
> 数据来源与核实状态：业务背景取自附录 G.1；关键数值（CpK≈0.82、0.95²⁰≈0.36、约束硬化 58%→78%、误差预算「千分之三」、夜班来料字段缺失沿 DMAIC 转一圈）已逐一对作者本地八审清样底稿核实（总论 0.1、第 3/4/8/16/17/21 章；底稿未随仓发布），核实记录见文末「数值核对台账」（台账随仓发布，外部用户可直接复核）。本文与构建期蓝图 case_8d 字段及附录 G.1 对齐，不引入未核实数值。
>
> **性质声明**：本案例为方法论**贯穿示范**，业务情节与数值为说明性示例（已对书内方法论锚点核实），非某次真实生产 8D 的记录。

## 0. 案例基座：这个 Agent 到底要做什么

A 产线转子组件出现尺寸超差，良率连续走低。质量工程师真正雇这个 Agent 做的，不是「陪聊」，而是一份**合规、可直接下发执行、可挂起签字的 8D 整改报告**，外加把人从「在 MES / ERP / 历史失效记录之间反复穿梭对数」的隐性焦虑里解脱出来。

把这件事拆成真正要走的步骤，就是附录 G.1 给出的**九步骤链**（落到工具调用一路展开约「二十来步」）：

```
① 从 MES 取该批次尺寸测量数据
② 用确定性代码算过程能力（CpK）/ 控制图判异
③ 检索失效模式库（FMEA）
④ 调取历史 8D 案例
⑤ 交叉比对设备保养 / 来料批次 / 环境记录
⑥ 生成根因假设
⑦ 起草 8D 整改报告
⑧ 触发不良物料冻结
⑨ 挂起等质量经理审批签字
```

这条九步骤链是后面每个阶段反复回到的同一根脊柱。请记住三个会被反复点名的步：第 ② 步（算 CpK，确定步）、第 ⑤ 步（交叉比对来料批次，全链最脆的约束步）、第 ⑥ 步（生成根因假设，全链方差最大的步）。

## 1. P0 立项分诊与可靠性认知（Define）

**路由子 skill**：`enterprise-agent-assessment`

**本阶段落点**：

- **建立 R≈pⁿ 主线**：把九步骤链落到工具调用约二十步，假设单步可靠度 p≈0.95，则任务整体可靠度 `R_task = 0.95²⁰ ≈ 0.36` —— 跑完二十步整体只有约 36% 概率全程无误，近三分之二的诊断会在某一步悄然出错。这是全书入口锚点。

  > **术语红线（必须照写）**：`R≈pⁿ` 只是便于建立直觉的**乐观基线**。真实写法是**条件概率连乘** `R = ∏ p_i`，其中 `p_i = P(step_i 正确 | 上游状态、上下文长度、噪声密度、既有错误)`。一旦形成错误的根因假设，模型会在错误前提上继续生成（self-conditioning 式误差传播），真实衰减往往比 pⁿ 更陡。绝不可把它简化成纯 pⁿ。

- **变差基线与作业链拆解**：把大模型当成一台「变差极大的信息处理器」做稳定性 / 偏差 / 一致性评估（像量具校准）；产出「大模型变差评估报告」与「任务作业链拆解图」（标出每步是否不可逆、是否需挂人工门）。
- **容差带定位**：本案例属表 3-2 中**制造质量签署**一档，可容忍错误率 **≤ 0.1%**（放行不合格批次召回损失巨大），对应最高护栏等级（三层控制面 + 过程能力硬校验 + 质量主管审批）。
- **圈出最脆瓶颈步**：立项当天就标出「**交叉比对推断根因**」（即第 ⑤→⑥ 段）为最脆环——答案不唯一、依赖推理、吃上游数据完整性。R≈pⁿ 的连乘下限由最低 p 的那一步主导，必须优先硬化它。

**Gate**：流程就绪度须达第③档（标准化：SOP 完整 + 异常路径明确 + 数据已 API 暴露）才有最低立项资格；六维评审任一维触红线即否决或强制加门；十二项否决条件每项挂负责人与完成时点；本任务消费外部内容（来料台账等），立项结论须显式标注「此任务涉及不可信输入」，移交正交轴 II。

**交付物**：`assessment-report.md`（GO / CONDITIONAL GO / NO-GO 裁决）+ 变差评估报告 + 作业链拆解图 + 不可逆动作清单 + 「控制档位 / 人工门位置 / 瓶颈步」三项结论（作为放行凭证）。

## 2. P1 三层控制面设计与任务分诊（Design 总骨架）

**路由子 skill**：`agent-control-plane-design`

在模型外盖**三层物理隔离的控制面**（不靠模型配合、靠外部代码强制）：

- **指令面（Instruction）**：把散文升级为 API 契约。8D 报告强制为含**八个固定字段**的 JSON（D1 组队 … D8 总结表彰），「第五个字段是空的」用一行 Schema 校验当场抓住打回；进度只能翻转指定状态字段，其余锁死（让错的动作做不出来）。
- **信息面（Information）**：Need-to-Know 按需供给 + **动态折叠（Dynamic Folding）**。分阶段开窗——意图分类只看当前警报，生成 SQL 只看表结构（绝不给数据行），撰写报告才注入过滤后证据。把上游「算过程能力」那三千字推理折成一行结论 **「过程能力指数 ≈ 0.82，状态：critical」**，输入从三千词元压到不足五十。
- **运行治理面（Harness Engineering）四步闭环**：状态机外置（JSON 状态锁字典 `{"MES数据已取":false, "过程能力已算":false, ...}` 初始全 false，某步产物经外部代码独立验证通过才翻 true）→ 权限隔离 → 强制校验熔断 → 人工审批。

**任务分诊**：同一个 8D Agent 内部分三档控制——

| 步性质 | 本案例例子 | 步数 | 落点 |
|---|---|---|---|
| 查表 / 算式（确定） | 算 A 批次 CpK | N≈1 | 一行代码，模型都不必碰 |
| 有套路（少数几步） | 按工序号精确命中失效模式库 | 几步 | 指令面契约 + 轻校验 |
| 真复杂诊断（长链条） | 「为什么连续三周良率下滑」 | 二十来步 | 完整运行治理面 + 必要时慢外环 |
| 不可逆执行 | 触发停机 / 发召回公函 | — | 强制人工门 |

铁轨钉在「触发停机 / 发送召回公函」这类强不可逆动作前，挂质量主管签字；权限边界物理锁死，Agent 只能起草可逆草稿。人工门按**可逆性**放，与模型置信度**彻底解耦**。

**Gate**：输出过 Schema 缺字段不放行；写 SQL 只给表结构不给数据行；不可信外部内容隔进数据区；状态外置 JSON 锁字典；季度护栏审计（触发率为零 / 误报率过高标「待评审」，防护栏膨胀）。

**交付物**：`control-plane-spec.md`（指令面 Schema 契约 + 信息面可见性矩阵与折叠规则 + 运行治理面状态字典模板与审批网关）+ 任务分诊漏斗判据表。

## 3. P2 构建·外部接入与信息控制面（快内环输入治理）

**路由子 skill**：`agent-control-plane-design`（接入层）+ `agent-guardrails-config`（信息洗净 / 熔断）

- **MCP 接三台只读服务器**：MES（取实时测量与设备数据）、ERP（核料批次与供应商记录）、沉淀多年的**失效模式库**（检索同工序历史失效与整改）。老路 N×M = 三套面条三种认证；改 MCP 后各包成一台只读服务器对外说同一种语言（N×M → N+M）。老旧失效模式库不重写，只**薄包一层翻译**并配短期只读凭证。三台全顾问型只读，自由穿梭积累证据；唯一「冻结物料」写动作锁进人工网关。
- **信息控制面五道工序**：召回 → 过滤 → 排序 → 消解 → 熔断，每道由代码把守。「先算数学，后比语义」：发起向量匹配前先用确定性 SQL 按硬指标（RPN 风险等级 / 版本号 / 有效期 / 权限）一票否决。引文信用分三维（信源权威 / 时效衰减 / 交叉验证）。检索置信度低于阈值，编排层直接拦截，强制返回结构化弃权 `{"status":"knowledge_insufficient","missing_info":"…"}`。
- **把根因检索的 p 从 0.7 洗到 0.95**：这正是第 1 章「张冠李戴」反例（第 ④ 步召回十几份相似旧 8D、第 ⑥ 步合并两份不同失效模式）的工程解药。洗输入是在每步入口悄悄抬地板，沿 pⁿ 复利。

**Gate**：网关参数校验拦下未注册参数；入库强制版本号 + 有效期、检索只返当前有效版；跨域向量空间隔离；外部来源裹 untrusted 块、块内指令不执行只当数据；高风险场景不能同模型既生成又裁决证据。

**交付物**：外部能力地图（顾问型 / 执行型白名单）+ MCP 服务器注册清单 + MCP 兼容性矩阵 + 信息控制面流水线规格 + 分级检索分诊表。

## 4. P3 构建·代码沙盒计算控制面（快内环计算治理）

**路由子 skill**：`agent-control-plane-design`

第 ② 步「算 CpK」是确定步，**绝不让模型自由生成最终数值**（律考前 10% 的模型会把 500 行 Excel 算错平均值——CpK 有唯一公式，必须可复验、可重跑）。完整作业：

1. **意图分解与代码生成**：Agent 知道自己无权直接「猜」CpK，生成一段 Python，用 Pandas 读 MES 传感器日志套统计公式算 CpK（算力归代码）。
2. **执行与自愈**：沙盒首次执行抛 `ValueError: cannot convert float NaN to integer`（数据有缺失值），驱动层把堆栈喂回模型，模型补一行 `df.dropna()`，第二次执行成功。算 CpK 是纯计算、天然幂等，重试无副作用。
3. **计算与解释合流**：沙盒算出 **CpK = 0.82**，并画正态分布控制图与特征重要性条形图。把 0.82 交回模型，模型依据预置基线给出业务解读：「实际 CpK 为 0.82，远低于 **1.33** 的安全基线，说明 A 产线加工尺寸存在严重系统性偏移；结合归因，主轴震动频率是核心异常因子，建议立即排查主轴轴承磨损。」

> **确定 / 随机分界（照写）**：`0.82` 这个数从头到尾没经过模型生成，是代码算的、可被审计一行不改地重跑的；而「0.82 意味着严重偏移、主因在主轴」这个**判断**才是模型承担的部分。三个月后审计来问「这份报告里的 0.82 是怎么来的」，工程师能调出当时那段代码原样重跑。

**对 R≈pⁿ 的兑现**：抬 p（把自由生成 CpK 那一步从概率路径转入可验证执行路径，方差归零）+ 缩 N（删掉「让我想想 CpK 公式是……」那段纯随机独白）+ 获得可验证性（代码即证据，落进 trace）。

**Gate**：数值结果必须可复现、不停留在模型自由生成层；代码验证层（Schema 绑定 / AST 扫描 / 库白名单 / 断言 / 黄金样例 / 差分校验）；带副作用动作必须幂等（唯一业务键服务端去重）。

**交付物**：代码沙盒安全配置清单 + 确定步 / 随机步切分表 + 幂等核查清单。

## 5. P4 构建·多智能体与编排（快内环组织设计 + 焊成流水线）

**路由子 skill**：`agent-human-loop-design`（科层制制衡）+ `agent-control-plane-design`（编排引擎）

- **数字科层制三道工序**（物理隔离、读独立库）：
  - 工序一 **Data Agent**：监督者把机台号 / 时间戳派下去，Python 沙盒拉 MES 算过程能力判异，契约 `{过程能力:0.82, 状态:critical}` 回交（数值确定性代码执行校验，绝不模型自由生成）。
  - 工序二 **Knowledge Agent**：检索失效模式库，提取纠正措施，带引用来源。
  - 工序三 **Compliance Agent**：作为严格裁判读独立质量标准原文，逐条清点八个 D，**缺 D3（临时围堵措施）直接打回**附错误日志。
  - 反模式：让同一个模型「三遍投票」当冗余——共用同一模型 / 知识库的裁判会一起错（伪冗余）。生成者 ≠ 审查者，且不共用同一套知识库。
- **DAG 焊成流水线**：拓扑写死流转。两类失败两套管辖——**物理错**（如取 MES 撞 503 网关错）走引擎指数退避（带抖动）；**业务错**走模型自愈。早期版本曾把两类失败搞混（503 被原样塞回模型，模型误以为 SQL 写错反复改、十几轮无效重试致诊断迟二十分钟），是反面教训。冻结物料带 **idempotency_key**，重试 3 次只冻一次。

**Gate**：拆分收益须盖过沟通税；每个 Agent 独立上下文窗口只传 JSON 契约；契约带版本号、不兼容即熔断；不可逆动作 `waitForApproval` 物理挂起；每节点紧后挂独立校验门。

**交付物**：Agent 拆分决策表 + 生成者-审查者制衡设计规范（附共因失效自查表）+ 通信契约 Schema 注册表 + 数字科层制拓扑图 + DAG / 状态机定义 + 失败处理矩阵 + 生产级 JSON 状态字典模板（`checkpoint_hash` / `idempotency_key` / `schema_version`）。

## 6. P5 设计期质量内建复核（CDOC Capability 预演）

**路由子 skill**：`enterprise-agent-assessment`

用 CDOC 把质量造进图纸——

> **术语红线**：CDOC = **Concept – Design – Optimize – Capability**，末阶段是 **Capability（能力）**，不是 Control。面向六西格玛设计（DFSS）的经典四阶段实现路线，源自 SBTI。早期个别培训资料末阶段曾写作 Control，成熟体系统一用 Capability。

- **五条 must-be → 五条 CTQ**：取数批次匹配、过程能力计算正确、案例不串失效模式、8D 八字段完整、不可逆动作不自动执行。
- **Concept 的价值**：A 团队跳过 Concept 上线，第一周把上批次记录当本批次根因、报告颠倒却条理分明无报错；B 团队三天做 Concept，同样的错误在发生那一步就被熔断拦下。
- **接口契约排除「沉默的单位错配」**：MES 尺寸数据有毫米 / 微米之分，单位列必填枚举，对不上即熔断。
- **TRIZ 分离化解假矛盾**：用空间分离让「算 CpK 不再由模型产生」，自治（模型自己决定算不算、自己写代码、自己读数、自己出报告）与可验证（数字是确定代码算的）两不耽误。

**Gate**：危险不可逆动作从动作空间删除（非提示词约束）；能力门真值集全程留出 + 标准答案独立确认 + 按比例纳入边界 / 异常 / 数据缺失疑难案例；**未过能力门不许上线**。

**交付物**：CDOC 四阶段设计文档（Concept must-be 清单 + CTQ 表 / Design 收敛动作空间 + 接口契约 + 数据契约 / Optimize 权衡矩阵 / Capability 真值集规格 + 能力门红线）。

## 7. P6 三道防线护栏（治理篇起点 Safety-I + 记忆权限）

**路由子 skill**：`agent-guardrails-config`（第一 / 三道防线）+ `agent-human-loop-design`（第二道防线可逆门）

> **章节红线（必须照八审真实结构）**：第 13 章 = 第一道防线·结构化护栏（Safety-I，IQC / IPQC / OQC）；第 14 章 = 第二道防线·人工复核可逆门；第 15 章 = 第三道防线·组织级记忆 / 权限隔离 / 知识代谢。

- **第一道防线（第 13 章·Safety-I）**：8D 报告的八个 D 字段**缺一个就熔断**（一行校验代码当场抓住打回，Safety-I 最纯粹的样子）。输入护栏还能拦下招聘经理「优先某档次院校、三十岁以下」这类好心却违法的指令并剥除。对抗性走**正交轴 II**：把外部内容当数据非指令；拆掉**致命三联**（同一 Agent 同时握有读私有数据 + 接触不可信外部内容 + 对外发送通道）至少一环；号称拦九成五的护栏对安全仍不够。
- **第二道防线（第 14 章·可逆门）**：A 产线转子尺寸超差，Agent 生成符合汽车行业标准的 8D 草案，建议「产线停机校准 + 向供应商发召回公函」（强不可逆，代价数十万元）。四步闸落下：状态机拦截写 ERP 的 API → 推送含 MES 控制图 / 历史故障 / 过程能力指数的审批卡片 → 质量主管把「立即停机」改为「今晚换班时停机」（隐性规则：高峰期非致命缺陷维修避开主生产班次）→ 凭数字签名解冻写入 ERP / MES。这次人工修改**被 diff 捕获，沉淀为判例种子**。
- **第三道防线（第 15 章）**：B 产线工程师问「参考隔壁 A 厂区（生产涉密零件）最近的良率和解决思路」——被包装成协助请求的隐性越权，在数据库引擎层校验发起人身份令牌**直接拦下**。组织级记忆分层、知识代谢（事件驱动 + TTL + 夜间聚类，代谢率健康区间 5%–15%）。

**Gate**：三道关一道不能省；八个 D 缺一即熔断；致命三联拆掉至少一环；不可逆动作「与置信度解耦永远挂起」；数字员工权限不高于唤醒它的人。

**交付物**：三层护栏配置清单（IQC / IPQC / OQC）+ 权限模型矩阵 + 输出风险分级响应表（A 软 / B 挂 / C 熔断）+ 熔断黑匣子日志模板 + 判例库 Schema（`scenario_tags` / `diff_summary` / `correction_reason` / `status` / `ttl_expiry`）。

## 8. P7 独立测量总线接通（慢外环地基 Measure）

**路由子 skill**：`agent-observability-setup`

8D Agent 上线后任务完成率 / 平均耗时 / 护栏拦截次数全绿，唯独「**根因判定方向对不对**」从未被单独测量，隐患漂移近三周（某工序根因被系统性归咎于来料波动，真实原因在设备保养），直到资深技师偶然翻报告才察觉。

补救：不再信子任务状态字段，定期抽取**真实交付报告**让独立审查逐份核对根因证据链（从看转述改为看真产物，漂移曲线才显形）。独立性做成两套库：生成靠相似性检索，审查挂载从工艺标准 / 安全规程固化的硬约束规则。

> **不可妥协**：测量必须独立于被测对象——让 Agent 给自己诊断打分等于自证清白。这是连接快慢两环的独立测量总线在 8D 案例里的落地。

**Gate**：测量口径必须与交付口径一致否则静默逃逸；绝不接受 Agent 给自己打分作质量证据；判断完成必须核对真产物（现地现物）；trace 存真实物非转述；打算当真理源的测量装置自身先过 MSA「它准不准」检验。

**交付物**：测量维度清单（该测的 Y / 后果权重 / 是否在测 / 由谁测）+ 独立审查机制设计 + 审查者标定报告 + 反例驱动评估集 + 细粒度 trace 规范。

## 9. P8 可观测性·约束·误差预算（慢外环 Analyze / Improve）

**路由子 skill**：`agent-observability-setup`

> **章节红线**：第 16 章 = 找瓶颈步（TOC 约束）；第 17 章 = 误差预算与优雅降级（Safety-II）；第 18 章 = 独立测量总线；第 19 章 = 可观测性。

- **可观测性定位（第 19 章）**：张冠李戴的 8D 报告事后调 Trace，沿执行树回走定位到第 ④ 步召回放太宽（证据污染）、第 ⑥ 步把两份不同失效模式案例合并为同一根因。
- **TOC 找真约束（第 16 章）**：用「首次走偏归因占比」这把尺子，不看利用率、不看告警密度。故障归因分布表显示：

  | 步骤 | 该步局部正确率 | 资源占用 | 首次走偏归因占比 | 是否约束 |
  |---|---|---|---|---|
  | 历史案例检索（最忙） | 高 | 高（最吵） | 8% | 伪瓶颈 |
  | 根因假设生成（最被投诉） | — | — | 11% | 否（多为上游污染传导 / 替上游挨骂的出口） |
  | **交叉比对设备 / 来料批次** | **70%** | 低（最安静） | **71%** | **是 ← 真约束** |

  最忙的不是瓶颈，最安静的那一步才是。用确定性代码（按时间戳和批次号精确 join）把这个约束步**从 0.70 硬化到 0.95**，整体合格率**从 58% 跳到 78%**（≈ +0.20，一步顶六十步）。随后**约束跳转**——当初的伪瓶颈检索步（约 96%）成为新约束。
- **误差预算（第 17 章·Safety-II）**：把「允许失败多少」当成一笔可花的账，定下「诊断结论的**逃逸率不得超过千分之三**」。预算按失败贡献**不均匀分配**：第 ② 步代码算 CpK 几乎不分预算、别镀金；第 ⑥ 步生成根因假设（全链方差最大）是瓶颈，预算重点压（独立审查 + 证据强制）。优雅降级真实软失败：某天上游 MES 小版本升级，第 ① 步取数多出一个从未出现的陌生字段，「字段完整性」前置校验（Safety-II 识别出这是意外）触发，退到降级阶梯第二级，只用能可靠解析的字段产出标注「含未知字段、本批结论待人工确认」的部分结果。

**Gate**：测量独立性门（评估器独立于生成器，禁自证）；告警仅当点穿控制限或连续同向漂移；动手大改前先临时无害化嫌疑约束、确认整体合格率应声跳升；预算未超可继续试点 / 消耗剩余额度，逼近或越线则激进动作自动冻结；降级默认值倒向「不认识就降级」绝不「不认识就放行」。

**交付物**：观测信号基线表 + 四象限运营看板规格 + 故障归因分布表 + trace 约束定位三步法 + 误差预算表 + 显式降级阶梯（完整 → 标注不确定 → 只读 / 部分 → 转人工 → 安全停机）+ 近失捕捉规则。

## 10. P9 评估与成本工程（慢外环计分与精益运营）

**路由子 skill**：`agent-observability-setup`

- **北极星 = 端到端成功率**（全部步骤正确 + 最终状态正确变更，即量 pⁿ 本身），拒绝单步平均与模型自报完成。
- **成本治理**：董事会上 CFO 问大模型账单算不清。早期让旗舰模型「读着数据自己算」过程能力指数（既贵又不稳）→ 改成确定性统计代码（**同时削成本又抬 p**）。模型路由：旗舰只负责中央规划 / 合规终审 / 复杂根因分析，轻量模型做抽 JSON / NL 转 SQL / 填模板。成本归因早期让模型读日志自估、同任务两次给两个数 → 改确定性代码从 trace 按词元 × 单价加总。

## 11. P10 DMAIC 外环判例飞轮（慢外环闭环 Control → Define）

**路由子 skill**：`agent-observability-setup`

> **章节红线**：第 21 章 = DMAIC 外环·判例飞轮。

**夜班来料字段缺失，沿 DMAIC 转一圈**（本案例最完整的慢外环演示）：

8D Agent 上线三周后，运维发现夜班提交的诊断单根因判定返工率**连续两周**系统性跳出波动带。先过「信号 vs 噪声」第一道闸——是信号，不是抖动，慢外环启动。

- **Define**：把偏差钉成可测 CTQ——「夜班根因返工率显著高于日班」，衡量指标是「按班次分层的根因一次通过率」，目标是把夜班拉到与日班无显著差异。最常见失误是跳过 Define 直接「未经验证地归因于模型能力不足、换个大模型」。
- **Measure**：去现场看真产物，不看汇总、不信转述（三现）。把夜班真实 trace 一条条拉出来，与日班并排比对，发现夜班出错的单子在第 ⑤ 步「交叉比对来料批次」时，拿到的**来料记录字段经常是缺的、空的**。仪表盘那个聚合数字只会告诉你「夜班有问题」，永远不会告诉你「问题出在第 ⑤ 步的来料字段缺失」。
- **Analyze**：把根因归属从模型能力转向**系统输入 / 数据源**。两班次用的是同一个模型、同一套提示词，区别只在输入数据质量——夜班来料记录录入有滞后，Agent 取数时记录尚未入库，拿着残缺证据把根因建在空字段上。**戴明 94/6**：约 94% 的问题源于系统、仅约 6% 源于个别特殊原因，改进应优先指向系统而非追责执行者（「修系统不指责模型」）。
- **Improve**：用最小有效的系统改动堵根因——在取数环节加一道**约 50 行的「来料完整性前置校验」**：第 ⑤ 步执行前先检查本批次来料记录完整性，字段齐了才往下走；不齐就触发优雅降级（标注「来料数据待补全」、退到部分结论、或挂起等数据补齐）。这道改进有双重身份：既是 **Safety-I**（把已能想到的失败模式提前写成自动拦截门），又把一段确定性完整性检查从模型随机链条里剥离（抬 p）。它设在**来料口（IQC 站位）**而非出厂口（OQC），是它有效的另一半原因。
- **Control**：把这道校验固化成快内环里一个**永久、机械执行的门禁**：写进代码、配上回归测试，任何未来改动碰掉它就立刻报红。改进若不固化只是一次走运，固化了才成资产。

**判例飞轮 C → D**：三周后又一张夜班单进来，场景相似。这一次在 Agent 构思根因之前，系统先从判例库召回那条「夜班来料字段易缺失 → 先验完整性」的判例，连同前置校验一起生效；字段已齐，Agent 根因一次判对。更重要的是——下一版需求评审时，「**来料批次完整性**」作为一条硬红线正式进了下一版 **CTQ 清单**。一次真实失败，从「一个被修掉的 bug」升级成「一条系统永远记得的规则」。

**Gate**：不 tampering 启动门（改前须证明该类错误系统性超出波动带）；Measure 真产物门；Analyze 先查系统门（戴明 94/6 先疑系统再疑模型）；Improve 一圈只解一个被锁定根因；Control 闭环焊门（代码门禁 + 回归测试）。

**交付物**：慢外环 DMAIC 改进 SOP（含信号 vs 噪声启动判据 + 五相检查表）+ 判例库 Schema 与代谢规范 + 改进验证模板 + 冷启动初始判例集。

## 12. P11 规模化生产准入（流水线终点 gate + 车队治理）

**路由子 skill**：`agent-production-gate`

> **章节红线**：第 22 章 = 从样板间到生产线·规模化落地与治理。

8D Agent 试点稳定后，全厂十几条产线一起上、兼管来料 / 出货抽检，三个月后失守于「没人说得清各分身版本 / 权限 / 是否接测量总线」。规模化 = **先盖好能管住几十份的治理底座**，再复制。

- **五步生产准入**（能力门**先于**流量门）：SOP 就绪性审查（oracle 先于实现）→ 沙盒压力测试（功能 / 对抗 / 韧性三类）→ **影子运行**（真数据、输出不执行，1–2 周）→ 灰度（5% → 20% → 50% → 100%，每档须有该档可靠性实测，任一档不达标立即回退）→ 全量后 30 天观察期。沙盒各项优异，但一进影子运行就暴露夜班根因判定与质量工程师系统性对不上（夜班来料字段录入滞后）。
- 该 Agent 能下达**产线全面停机 / 向供应商发召回公函** = 关键风险，须走完整五步。
- 一次飞轮转动把「**数据源完整性**」固化成下一版**五步准入 SOP 的强制检查项**。与总论 R≈pⁿ 主线在此收口：每次失败固化为永久准入红线，地板逐圈抬高。

**Gate**：五步准入全过才放流量；三权分立（业务 / 工程 / 安全合规一票否决）独立量具；外发动作白名单网关机械拦截；合规架构十项（算法备案 / 数据出境 / 运营 trace 留存 90 天、人工审批 trace 3 年 / 红队注入测试 / 季度复审 / 一键熔断演练）。

**交付物**：`production-readiness-report.md`（五道闸门检查表 + READY / CONDITIONAL / BLOCKED 裁决 + 阻塞项清单）+ 五步生产准入 SOP + 分级审批矩阵 + 中国合规治理清单（十项）+ 三组看板指标定义 + 全书 22 章上线自检单（附录 C）。
