## 逐条判据 10 条

### 1. QFD 结构完整性：部分

证据：有 VOC→CTQ 表、How 表、关系矩阵、技术重要度、屋顶、竞争评估、目标值、Phase 1.5/2 cascade，结构基本成型。

问题：

- How 表缺原判据要求的 `技术难度`、`风险等级` 字段。
- 目标与优先级表只列 HOW-01/02/03/04/05/07/08，漏掉 HOW-06、HOW-09、HOW-10、HOW-11。
- 竞争评估漏掉 HOW-07、HOW-10、HOW-11。
- Phase 3 没有自然 cascade，只提到 1.5/2。

### 2. VOC 质量：PASS

证据：

- VOC-01 “良率一掉我最怕停线等结论”、VOC-03 “别让我在 MES/ERP/SPC/历史8D 之间来回对批次、对口径”、VOC-05 “撤不回的事必须我点头”都是客户语言，不是方案语言。
- 覆盖良率下滑、8D、RCA、停机损失、跨系统查证、人工复核、逃逸风险。

小问题：客户角色覆盖了质量经理、QE、SQE、工艺工程师，但缺 IT/数据管理员、生产主管的直接 VOC；对数据权限、主数据维护、现场班组执行负担覆盖偏弱。

### 3. CTQ 质量：部分

证据：

- CTQ-02 “根因有证据支撑、可追溯、经得起复核”可转为 HOW-02/HOW-03。
- CTQ-05 “不可逆动作必经人工确认、与置信度解耦”质量约束明确。
- CTQ-06 “该报警不漏报、不静默逃逸”对应 HOW-05。

问题：

- CTQ-01 “在不停产或最短停机下给出可行动的诊断方向”不是严格 CTQ，缺验收口径；“可行动”未定义。
- CTQ-07 “同类问题不二次发生”超出 Phase 1 Agent 诊断可直接保证的范围，HOW-11 “判例复用率”只能支撑知识复用，不能证明“不二次发生”。
- CTQ-10 “投入产出可量化、价值可追溯”需要拆成停线损失节省、返工报废降低、人工工时节省等，否则业务价值口径偏泛。

### 4. 工程特性质量：部分

证据：

- HOW-01 诊断闭环时长、HOW-02 根因 Top-N 命中率、HOW-03 证据链覆盖率、HOW-04 字段对齐准确率、HOW-08 人工门覆盖率都是工程质量度量，不是功能模块。
- 多数 How 有定义、单位、方向、目标值、验证方法。

问题：

- HOW-05 单位写 `ppm`，目标写 `≤3‰`，单位不自洽；3‰ = 3000 ppm，不是 ppm 级目标。
- HOW-10 目标 “较人工提前 ≥N 批”不可验收，N 未给出。
- HOW-07 “合规字段完整率”只测字段齐全，不测 8D 内容质量、事实/假设/结论/建议分离准确率。
- 缺关键 How：数据缺失显性提示率、未证实结论拦截率、事实/推断/假设/建议分类准确率、纠正/预防措施可验证率。
- 原判据要求技术难度、风险等级，表中没有。

### 5. 关系矩阵有效性：部分

证据：

- 有区分度，不是全 9。比如 CTQ-05 只强连 HOW-08，CTQ-09 只强连 HOW-01。
- 加权公式自洽。抽查：
  - HOW-01 = CTQ-01 10×9 + CTQ-03 8×3 + CTQ-09 6×9 = 168，正确。
  - HOW-03 = CTQ-02 9×9 + CTQ-04 9×3 + CTQ-06 9×3 = 135，正确。
  - HOW-05 = CTQ-06 9×9 + CTQ-02 9×3 + CTQ-08 7×3 = 129，正确。

问题：

- 完全没有使用弱关系 `1`，矩阵只有 9/3/空，区分度还不够细。
- CTQ-04 8D 合规下发对 HOW-03 证据链只给 3 偏低；8D 客户审核通常强依赖证据链。
- CTQ-10 价值可量化只连 HOW-06 和 HOW-09，漏掉 HOW-01 诊断闭环时长、HOW-05 逃逸率、HOW-10 提前预警，这些都直接影响停线/报废/客诉成本。
- HOW-07、HOW-08 都只有单一强关系，作为硬门可以接受，但矩阵排序会低估它们，需要在表内明确“硬门不参与线性排序”而不是只在结论里补充。

### 6. 屋顶相关性深度：部分

证据：

- 识别了真实冲突：HOW-01 时长 vs HOW-03 证据覆盖，HOW-01 时长 vs HOW-05 逃逸，HOW-09 工时下降 vs HOW-08 人工门覆盖。
- “自动化 vs 人审可控按可逆性分轨”是有效质量工程判断。

问题：

- 原判据要求正/负相关，结果只列负相关，没有正相关。比如 HOW-03 证据链覆盖率与 HOW-06 8D 一次审核通过率应为正相关；HOW-04 对齐准确率与 HOW-03 证据覆盖也应为正相关。
- 屋顶第 6 行 “06 报告自动化↑”不是 HOW-06。HOW-06 实际是“8D 一次审核通过率”，被错误替换成“报告自动化”。
- “专家可编辑性”不是已定义 How，却进入屋顶矩阵，破坏 HOQ 对象一致性。
- 部分处理建议已经进入 Phase 2 解法，如“缓存、重计算下推、接入层解”，可以作为设计含义，但不应喧宾夺主。

### 7. 技术竞争评估：部分

证据：

- 比较对象符合要求：当前人工、传统 MES/BI、通用 LLM、目标 Agent。
- 维度基本按 How 评分，不是泛泛营销。

问题：

- 漏评 HOW-07 合规字段完整率、HOW-10 漂移检测提前量、HOW-11 判例复用率。
- 评分缺度量基线，例如 HOW-01 人工=2、目标=5 没有映射到 8-24h vs ≤2h；HOW-05 人工=3、目标=5 没有逃逸率数据。
- “目标 Agent”多项直接给 5，风险是把目标当结果，没有说明达标条件。
- 通用 LLM “不接数据”不是稳定假设；应按“未受控、未集成、未验证”的默认形态描述。

### 8. 目标值与验证性：部分

证据：

- HOW-01 ≤2h、HOW-02 Top-3≥90%、HOW-03 ≥95%、HOW-04 ≥99%、HOW-06 ≥85%、HOW-07 100%、HOW-08 100% 有方向和验证方法。
- 明确 “待测基线需试点回填”。

问题：

- HOW-10 “≥N 批”不可测试。
- HOW-05 ppm/‰ 单位冲突。
- 多数当前基线为“待测”，可作为 Phase 1 假设，但不能形成完整验收底稿；至少应列人工/现有系统预估基线或标注“无基线，必须先测”。
- 目标与后续部署表缺 HOW-06/09/10/11，导致目标值没有完整落到优先级。
- 技术难度、风险等级缺失，违反原 matrix_spec。

### 9. 质量工程一致性：PASS_WITH_MINOR_FIXES

证据：

- 正确使用 VOC、CTQ、Kano、8D、RCA、证据链、审核、挂签、逃逸、人工门、Must-be 硬门。
- 对不可逆动作、人审、静默逃逸、数据缺失风险的保守性到位。
- 没有退化成普通聊天机器人或报表助手。

问题：

- 8D 术语仍偏“报告合规”，缺围堵措施、根因验证、纠正措施、预防措施、效果验证这些 8D 质量点。
- 原约束要求“事实、推断、假设、建议区分”，实现里没有对应 CTQ/How。
- HOW-07 只测字段完整，不能代表 8D 质量体系文件合规。

### 10. 可执行性：部分

证据：

- 可以作为 Phase 1 工作底稿继续展开；Top How、硬门、关键冲突、待测假设都列出来了。
- 能支撑 Phase 1.5：例如 HOW-01/02/03/05/08 可继续展开到诊断、证据、验证、人审控制。

问题：

- 缺完整验收字段：技术难度、风险等级、完整目标表、完整竞争评估。
- 若直接进入 Phase 1.5，会把几个缺口带下去：事实/假设/结论/建议分离、数据缺失显性提示、8D 纠正/预防措施质量、HOW-10 目标未定。
- 屋顶里混入未定义 How，会导致下一阶段 cascade 对象不干净。

## 加分项

- 明确区分“一票否决硬门”和线性加权优先级，尤其 HOW-05/07/08 没有被低排名误伤。
- 技术重要度计算透明，抽查正确。
- VOC 写得像现场人员会说的话，不像产品经理功能清单。
- “不可逆动作人工门与置信度解耦”是很强的质量工程判断。
- 对通用 LLM 的短板定位准确：证据、逃逸、人审门，而不是泛泛说“更智能”。

## 总体 verdict

**PASS_WITH_FIXES**

一句话总评：Claude 的 Phase 一 HOQ 主结构和质量工程方向是对的，但目标/风险字段不完整，屋顶混入非 How，部分指标不可验收，不能无修正直接进入下一阶段。

必改项：

1. 补齐 How 表和目标优先级表的 `技术难度`、`风险等级`、完整 HOW-01 至 HOW-11。
2. 修正 HOW-05 单位与目标，统一 ppm 或 ‰。
3. 给 HOW-10 设定具体 N 值或改成可验证的时间/批次数目标。
4. 补入事实/推断/假设/建议分离、数据缺失显性提示、未证实结论拦截相关 How。
5. 修正屋顶：只允许已定义 How 进入；补正相关；删除或重写 “06 报告自动化↑ / 专家可编辑性” 这类非 How 项。
6. 补全竞争评估 HOW-07/10/11，并把 1-5 分连接到可度量口径。
7. 扩展 8D 合规 CTQ，不只测字段完整，还要覆盖围堵、纠正、预防、验证、签核与证据链质量。
